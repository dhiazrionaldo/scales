# AI-Powered Location Suggestion System

**Overview**: Intelligent warehouse location recommendations powered by Claude AI

---

## Architecture

The location suggestion system uses Claude Opus 4.6 to analyze incoming pallets and recommend optimal warehouse locations based on:

1. **Pallet Characteristics**
   - Product name
   - Quantity
   - Weight
   - Batch number
   - HU label

2. **Warehouse Capacity**
   - Current occupancy per location
   - Available slots (max_stacks - current_occupancy)
   - Location status (AVAILABLE, FULL, MAINTENANCE, DISABLED)

3. **Warehouse Layout**
   - 2D grid positioning (zone, row, column)
   - Zone-based organization
   - Proximity metrics

4. **Warehouse Efficiency Goals**
   - Maximize space utilization
   - Group similar products in zones
   - Minimize travel distance
   - Balance load distribution

## System Components

### 1. Location Suggestion Service

**File**: `src/lib/ai/location-suggestion.service.ts`

**Responsibility**:
- Orchestrate Claude API calls
- Parse AI-generated suggestions
- Provide rule-based fallback (when AI unavailable)
- Calculate distance scores

**Key Methods**:

```typescript
async suggestLocations(request: LocationSuggestionRequest): Promise<LocationSuggestion[]>
```

Takes:
- `pallet`: HU label, product, quantity, batch, weight
- `availableLocations`: All warehouse locations with occupancy data
- `warehouseId`: Warehouse identifier
- `companyId`: Company identifier (multi-tenant)

Returns: Top 5 location suggestions ranked by AI

### 2. Receiving API Endpoint

**File**: `src/app/api/finish-good/receiving/route.ts`

**Workflow**:
1. OCR extraction → Pallet data
2. Duplicate detection
3. Pallet registration → Database
4. **NEW: AI Location Suggestion** ← Claude API call
5. Response with suggested locations

## How It Works

### Claude's Decision Process

The system sends Claude:
- **System Prompt**: Warehouse logistics expertise context
- **Pallet Data**: Incoming pallet specifications
- **Location Inventory**: All available locations with capacity

Claude returns JSON with ranked suggestions:

```json
[
  {
    "location_code": "A-001",
    "reasoning": "Zone A has similar products, adequate capacity, and proximity to receiving area"
  },
  {
    "location_code": "B-003",
    "reasoning": "Available slots, good grid proximity, less congested zone"
  },
  ...
]
```

### Suggestion Mapping

The service:
1. Parses Claude's location codes
2. Maps to full location objects
3. Calculates capacity percentages
4. Computes distance scores
5. Returns top 5 in order

### Response Format

Each suggestion includes:

```typescript
interface LocationSuggestion {
  location_id: string;           // UUID for database
  location_code: string;         // "A-001" (human-readable)
  capacity_available: number;    // 0-100% (percentage of available slots)
  distance_score: number;        // 0-1 (proximity metric)
  reasoning?: string;            // Why Claude suggested this (optional)
}
```

## Fallback Behavior

If Claude API is unavailable:

1. **Service Error**: Falls back to rule-based suggestion
2. **API Error**: Returns empty suggestions, then basic logic
3. **Parsing Error**: Uses pre-defined heuristics

### Rule-Based Fallback

When AI is unavailable, the system uses:

```typescript
// Sort by:
1. Available capacity (descending)
2. Grid distance (ascending - closer to 0,0)
3. Limit to 5 locations
```

This ensures the receiving process never fails due to AI unavailability.

```
    ↓

    ↓

    ↓

    ↓

    ↓

    ↓

    ↓
Create stock_movement record
    ↓
SUCCESS: Pallet stored
```

## Performance Considerations

### Latency

- **Claude API call**: ~500ms - 2s (network + inference)
- **Supabase queries**: ~50-100ms each
- **Total time**: ~2-3s for location suggestion

### Cost

- **Opus 4.6 pricing**: $5 per 1M input tokens, $25 per 1M output tokens
- **Per request estimate**: ~0.0001-0.0002 USD per pallet (~$0.00015)
- **Monthly (1000 pallets/day)**: ~$4.50/month

### Token Usage

- **Average tokens per request**: 300-400 input, 50-100 output
- **Batch**: All locations data + pallet info

**Cost Optimization Options**:

1. **Prompt Caching**: Cache warehouse locations data (if static during shift)
2. **Batch API**: Process location suggestions asynchronously (24-hour batch)
3. **Model Downgrade**: Use Sonnet 4.6 for 40% cost savings, similar quality

## Troubleshooting

### Issue: "API key not found" or 401 errors

**Solution**: Ensure `ANTHROPIC_API_KEY` is set in environment.

```bash
# Check env var (in terminal)
echo $ANTHROPIC_API_KEY

# Should output: sk-ant-...
```

### Issue: Slow location suggestions (> 5 seconds)

**Possible causes**:
- API latency spike (check Anthropic status page)
- Network issue
- Many locations in warehouse (large prompt)

**Solution**: System will timeout and use fallback.

### Issue: Claude returns invalid JSON

**Solution**: Service validates and uses fallback if parsing fails.

Check logs:
```
[AI-LOCATION-SUGGESTION] Could not parse JSON from response
```

### Issue: Location suggestions seem arbitrary

**Solution**: Review Claude's reasoning in logs and adjust system prompt.

Edit `src/lib/ai/location-suggestion.service.ts` → `systemPrompt` variable.

## Security & Compliance

### Data Privacy

- **No PII Storage**: Pallet data is not stored in Anthropic's systems (processed in real-time)
- **Company Isolation**: Each company_id filters data independently
- **No Model Training**: Data is not used for Claude model training

### API Security

- API key never exposed in frontend code
- Always use server-side API routes
- Environment variables are server-only

### Audit Trail

All location assignments are logged in `stock_movements` table:

```sql
SELECT * FROM stock_movements
WHERE movement_type = 'INBOUND'
ORDER BY created_at DESC;
```

Tracks:
- Which pallet
- Which location
- Which operator
- When it happened

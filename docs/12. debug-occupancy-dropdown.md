# Debug: Occupancy Dropdown Not Updating

**Guide to identify why dropdown shows 100% free despite updated database**

---

## Diagnosis: Check Server Logs

The issue is that **the API is returning wrong capacity_available data**. Enhanced logging will show which query path is being used.

### Step 1: Check Server Logs

When you receive a pallet, look at your server terminal for messages like:

```
[RECEIVING] Fetching locations for warehouse: abc-123, company: xyz-789
[RECEIVING] Found X locations, requesting AI suggestions...
[RECEIVING] Claude AI suggested X locations
```

**OR if using fallback**:

```
[RECEIVING] ⚠️ location_occupancy view failed, falling back to warehouse_locations
[RECEIVING] Fallback - A-001: 2/4 (50.0%)
[RECEIVING] Fallback - A-002: 0/4 (100.0%)
```

This tells you:
- ✅ If using view (correct path)
- ⚠️ If using fallback (view not available)
- 🔴 If errors occur during location queries

### Step 2: Check Browser Console (F12)

Look for:

```
[RECEIVING] API Response: {
  suggested_locations_count: 5,
  suggested_locations: [
    { location_id: "...", location_code: "A-001", capacity_available: 50, ... },
    { location_id: "...", location_code: "A-002", capacity_available: 100, ... }
  ]
}
```

**If capacity_available is wrong here**, the API calculation is broken.

## Common Issues & Fixes

### Issue 1: Using Fallback Path (View Not Available)

**Log shows**:
```
[RECEIVING] ⚠️ location_occupancy view failed
```

**Cause**: View doesn't exist or RLS is blocking it

**Fix**:
```bash
# Option 1: Re-apply the view fix
# See docs/10. fix-location-occupancy-view.md

# Option 2: Check if view exists
SELECT * FROM information_schema.tables
WHERE table_name = 'location_occupancy'
AND table_schema = 'public';
```

Should return 1 row (the view)

### Issue 2: Fallback Calculating Wrong Occupancy

**Log shows**:
```
[RECEIVING] Fallback - A-001: 0/4 (100.0%)
```

**But database shows**: current_occupancy = 2

**Cause**: Fallback query might be counting wrong statuses

**Fix**: (Already applied in latest code)
- Fallback now counts both `PUTAWAY_PENDING` and `STORED` statuses
- Make sure you're running the latest code

### Issue 3: Claude AI Returning Wrong Data

**Log shows**:
```
[AI-LOCATION-SUGGESTION] Claude response received
[RECEIVING] Claude AI suggested 5 locations
```

**But capacity_available is 100 for all**:

**Check**: Location suggestion service calculations

```typescript
// In location-suggestion.service.ts
return {
  capacity_available: Math.max(
    0,
    ((location.available_slots || location.max_stacks) / location.max_stacks) * 100
  ),
};
```

Should calculate: `(available_slots / max_stacks) * 100`

## Step-by-Step Troubleshooting

### Test 1: Verify Database View Works

```sql
-- Check view returns fresh data
SELECT location_code, current_occupancy, available_slots, occupancy_status
FROM location_occupancy
WHERE location_code = 'A-001';
```

Expected: Shows current occupancy from database

If returns 0 rows: View doesn't exist or has RLS issues

### Test 2: Verify Pallet Status

```sql
-- Check what pallets are assigned to location
SELECT id, status, location_id
FROM pallets
WHERE location_id = (SELECT id FROM warehouse_locations WHERE location_code = 'A-001')
ORDER BY created_at DESC
LIMIT 10;
```

Expected: Shows pallets with status `PUTAWAY_PENDING` or `STORED`

If shows RECEIVED: Pallets haven't been confirmed to location yet

### Test 3: Check Occupancy Calculation

```sql
-- Manually calculate what occupancy should be
SELECT
  location_code,
  max_stacks,
  (SELECT COUNT(*) FROM pallets p
   WHERE p.location_id = wl.id
   AND p.status IN ('PUTAWAY_PENDING', 'STORED')) as manual_occupancy,
  ROUND(
    (SELECT COUNT(*) FROM pallets p
     WHERE p.location_id = wl.id
     AND p.status IN ('PUTAWAY_PENDING', 'STORED'))::numeric / max_stacks * 100,
    1
  ) as manual_capacity_available
FROM warehouse_locations wl
WHERE location_code IN ('A-001', 'A-002', 'A-003')
LIMIT 10;
```

Compare with what UI shows. Should match.

### Test 4: API Response Check

In browser DevTools (F12):

1. Go to **Network** tab
2. Filter: type = `fetch`
3. Receive a pallet
4. Click on `/api/finish-good/receiving` request
5. Go to **Response** tab
6. Search for `suggested_locations`
7. Check `capacity_available` values

Should show updated percentages

## If Still Broken

Check in this order:

1. **View exists**: `SELECT * FROM location_occupancy LIMIT 1;`
2. **Pallets exist**: `SELECT COUNT(*) FROM pallets WHERE location_id IS NOT NULL;`
3. **Pallet status correct**: `SELECT DISTINCT status FROM pallets;` (should include PUTAWAY_PENDING)
4. **API logs**: Check server terminal for errors
5. **RLS policies**: Could be blocking view queries
6. **Database connection**: Is it stale?

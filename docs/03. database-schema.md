# SCALES - Lean Multi-Tenant Database Schema
### 1. Core Principles for AI Agents
* Mandatory Multi-Tenancy: Every table (except global system lookups) MUST include company_id.

* The Stack Constraint: Locations are defined by max_stacks. Occupancy is calculated by the number of pallets currently assigned to a location_id.

* Single-Action Audit: Every movement must be captured in a single stock_movements entry to ensure traceability.

### 2. Organization & Master Settings
These tables define the sister companies and the global configuration for the warehouse.
```
companies
id: UUID PRIMARY KEY

name: VARCHAR (e.g., "Sister Company A")

settings: JSONB (Store OCR formats, auto-confirm toggles, and logo)

created_at: TIMESTAMP
```
```
warehouses
id: UUID PRIMARY KEY

company_id: UUID REFERENCES companies(id)

name: VARCHAR

address: TEXT
```
### 3. Warehouse Layout (The Stack Model)
This table supports the "Layout Automizer" by defining vertical capacity.
```
warehouse_locations
id: UUID PRIMARY KEY

warehouse_id: UUID REFERENCES warehouses(id)

location_code: VARCHAR (e.g., "A-01")

zone: VARCHAR (e.g., "Zone A")

max_stacks: INTEGER DEFAULT 4 (Vertical limit)

status: VARCHAR (AVAILABLE, FULL, MAINTENANCE)
```
### 4. Inventory & Single-Action Pallets
To enable "Single-Action" speed, we merge HU data and Pallet status into one flat structure.
```
pallets
id: UUID PRIMARY KEY

company_id: UUID REFERENCES companies(id)

hu_code: VARCHAR UNIQUE (Scanned from label)

product_name: VARCHAR

qty: INTEGER

batch: VARCHAR

location_id: UUID REFERENCES warehouse_locations(id) (NULL if shipped)

status: VARCHAR (STORED, SHIPPED, DEFECT)

last_scanned_at: TIMESTAMP

created_at: TIMESTAMP DEFAULT NOW()
```
# 5. Movement & Reporting Audit
Required for the "Traceability" and "Movement Log" reports.
```
stock_movements
id: UUID PRIMARY KEY

pallet_id: UUID REFERENCES pallets(id)

from_location_id: UUID (NULL for new Inbound)

to_location_id: UUID (NULL for Outbound)

movement_type: VARCHAR (INBOUND, OUTBOUND, ADJUSTMENT)

operator_id: UUID REFERENCES users(id)

created_at: TIMESTAMP DEFAULT NOW()
```
### 6. Delivery Orders (Outbound Context)
Used to validate the "Scan-to-Ship" process.
```
delivery_orders
id: UUID PRIMARY KEY

company_id: UUID REFERENCES companies(id)

do_number: VARCHAR UNIQUE

status: VARCHAR (OPEN, COMPLETED)

items: JSONB (Expected product/qty for validation)
```
### 7. AI Agent Implementation Instructions
Occupancy Query: To calculate if a stack is full, the agent must query:
```
SELECT count(*) FROM pallets WHERE location_id = X AND status = 'STORED'.
```
* Safety Check: The agent must never allow an Inbound action if count(*) >= warehouse_locations.max_stacks.

* Cross-Tenant Leak: Always append AND company_id = auth.uid().company_id to every query.

You are right—if the AI agent doesn't see a clear definition for Auth and Roles, it might build a system where every user has "Super Admin" powers, which is dangerous for a multi-tenant setup with sister companies.

In your manufacturing environment, you likely only need two simple roles: Admin (who sets up the stacks and sees reports) and Operator (the person doing the "Single-Action" scanning).

### 8. Auth, Roles & Permission Tables
This structure ensures that "Admin" and "Operator" permissions are strictly enforced per sister company.
```
roles
id: UUID PRIMARY KEY

name: VARCHAR (e.g., 'ADMIN', 'OPERATOR')

description: TEXT
```
```
user_roles (The Role Manager)
id: UUID PRIMARY KEY

user_id: UUID REFERENCES auth.users(id)

company_id: UUID REFERENCES companies(id)

role_id: UUID REFERENCES roles(id)

Constraint: UNIQUE (user_id, company_id) — A user has exactly one role per sister company.
```
```
profiles
id: UUID PRIMARY KEY (References auth.users.id)

full_name: VARCHAR

last_active_company_id: UUID REFERENCES companies(id) — Remembers which sister company the user was last managing.
```
